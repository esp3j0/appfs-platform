# APPFS v0.2 真实 App 对接规范（骨架）

- 版本：`v0.2-draft`
- 状态：`Frozen (Phase A, 2026-03-20)`
- 依赖文档：
  - [APPFS-v0.2-总览.zh-CN.md](./APPFS-v0.2-总览.zh-CN.md)
  - [APPFS-v0.2-Connector接口.zh-CN.md](./APPFS-v0.2-Connector接口.zh-CN.md)
  - [APPFS-v0.2-合同测试CT2.zh-CN.md](./APPFS-v0.2-合同测试CT2.zh-CN.md)

## 1. 目标

1. 给真实 app 接入提供统一执行规范与验收路径。
2. 降低“只在 mock 里通过、真实环境失效”的风险。
3. 让接入过程可审计、可回归、可发布。

## 2. 对接范围

适用于：

1. IM/协作类应用
2. CRM/工单类应用
3. 文件与知识库类应用
4. 其他具备 API/SDK 能力的 SaaS 或自建系统

不适用于：

1. 无法提供稳定 API 或 SDK 的目标系统。
2. 无法满足最小鉴权与审计要求的目标系统。

## 3. 对接前置清单

1. 业务能力清单（动作、资源、实时性要求）。
2. 上游 API 清单（分页、查询、写操作、错误码）。
3. 认证方案（token、刷新、租户隔离）。
4. 速率限制与配额规则。
5. 数据分类与脱敏要求。

## 4. 节点建模规则

## 4.1 snapshot 节点

1. 面向有限数据，暴露为 `*.res.jsonl`。
2. 声明 `snapshot.max_materialized_bytes`。
3. 需要定义从上游分页/游标到本地 JSONL 的映射策略。

## 4.2 live 节点

1. 面向动态流式内容，暴露为 `*.res.json`。
2. 必须提供分页 envelope 与句柄生命周期。
3. 必须声明 `fetch_next` 与 `close` 控制动作路径。

## 4.3 action 节点

1. 统一通过 `.act` JSONL 提交。
2. payload 结构以 schema 为准。
3. 不允许原始文本直写。

## 5. 真实上游映射规范

1. 分页映射：
   - 上游分页 cursor/page/token -> Connector 统一 cursor。
2. ID 映射：
   - 上游 ID -> AppFS 资源路径分段（稳定、可逆或可追踪）。
3. 错误映射：
   - 上游错误 -> v0.2 统一错误码。
4. 时间戳与排序：
   - 明确增量同步顺序依据（ts、seq、version）。

## 6. 鉴权与安全

1. 连接器必须支持最小认证状态检查（`health`）。
2. 敏感字段必须脱敏后写入事件与日志。
3. 凭证生命周期要支持刷新与失效回退。
4. 任何越权访问必须映射为标准拒绝错误并无副作用。

## 7. 可靠性要求

1. 支持超时、限流、5xx 的重试建议输出（`retryable`）。
2. snapshot 扩容必须可恢复，避免半成品污染。
3. 同资源并发读取需去重，避免上游风暴。
4. 终态事件必须唯一。

## 8. 可观测性要求

最小字段：

1. `request_id`
2. `client_token`
3. `trace_id`
4. `upstream_request_id`（如有）
5. `connector_id`

最小指标：

1. 上游调用成功率
2. 限流触发率
3. 扩容延迟
4. 失败错误码分布

## 9. 测试与验收流程

1. 单元测试：
   - 映射逻辑、错误码映射、鉴权失败分支。
2. 合同测试：
   - 通过 CT2 required 子集（至少 `CT2-001`..`CT2-009`）。
3. 集成测试：
   - 在真实上游 sandbox 环境跑通关键业务流。
4. 发布前验证：
   - 回归 v0.1 baseline smoke，确保无非预期回归。

## 10. 迁移与上线策略

1. 新接入默认走 v0.2 Connector 路径。
2. 上线采用灰度策略：
   - 按 app、按租户、按资源类型分批。
3. 问题处置策略：
   - 暂停受影响 connector 的写入路径或降级受影响能力。
   - 以审计日志与失败证据驱动修复与重放。

## 11. 约束

1. 本文为接入规范骨架，具体 app 可附加领域规则。
2. 所有附加规则不得破坏 v0.2 核心协议语义。
3. 文档未冻结前，不进入真实 app 生产接入。

## 12. 验收

1. 新 app 接入时可按本规范逐项打勾完成。
2. issue 可直接按章节拆分，避免遗漏关键环节。
3. 能为 RC 审核提供可复用证据模板。
