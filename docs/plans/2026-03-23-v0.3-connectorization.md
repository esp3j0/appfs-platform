# APPFS v0.3 Connectorization Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Deliver AppFS `v0.3` connectorization with a single latest connector contract across `in-process`, `HTTP bridge`, and `gRPC bridge`, replacing the current mixed/skeleton snapshot path.

**Architecture:** Introduce a new breaking connector surface (`AppConnectorV2`) and rewire runtime read/action/paging paths to call connector methods directly. Transport-specific adapters become thin protocol shims over the same V2 contract. CI gates are updated to prove parity for required CT2 capability paths.

**Tech Stack:** Rust (`agentfs_sdk`, `cli` runtime), Python bridge servers, protobuf/gRPC, shell contract tests (CT2), GitHub Actions.

---

## 0. Scope And Rules

1. This is a **v0.3** line. Breaking trait/protocol updates are allowed.
2. `AppAdapterV1` and `bridge v1` are treated as legacy compatibility surfaces.
3. Runtime code must no longer rely on internal snapshot expansion stub paths for shipping behavior.
4. `in-process`, `HTTP bridge`, and `gRPC bridge` must expose the same effective connector semantics.
5. Platform-level completion is not sufficient; at least one real-app pilot must pass before claiming "ready for real integration".

## 1. Target Contract (V2)

### 1.1 New canonical connector surface

Create a new Rust trait (name to finalize in task 1, default: `AppConnectorV2`) with required methods:

1. `connector_id() -> ConnectorInfo`
2. `health(ctx) -> HealthStatus`
3. `prewarm_snapshot_meta(resource_path, timeout, ctx) -> SnapshotMeta`
4. `fetch_snapshot_chunk(request, ctx) -> FetchSnapshotChunkResponse`
5. `fetch_live_page(request, ctx) -> FetchLivePageResponse`
6. `submit_action(request, ctx) -> SubmitActionResponse`

### 1.2 Contract principles

1. Snapshot read-through is connector-owned upstream fetch behavior; runtime owns cache materialization/lifecycle.
2. Live pagination source-of-truth is connector response (`items + page`), runtime owns handle lifecycle.
3. Action submit semantics remain ActionLineV2-compatible (inline/streaming), but contract shape upgrades to V2 types.
4. Error mapping follows unified code set (`INVALID_PAYLOAD`, `TIMEOUT`, `RESOURCE_EXHAUSTED`, `NOT_SUPPORTED`, etc.).

### 1.3 Data consistency requirements (must-pass)

1. `fetch_snapshot_chunk` must define:
   - stable ordering key;
   - record-level dedupe key;
   - resumable cursor/offset semantics with no silent duplicates or drops.
2. `fetch_live_page` must define:
   - recoverable cursor semantics;
   - stable `page` envelope (`handle_id`, `page_no`, `has_more`, `mode=live`);
   - explicit error mapping for expired/invalid upstream cursors.
3. Runtime must not retain business-data snapshot stub fallback as a shipping success path.

### 1.4 Operational readiness requirements (must-pass)

1. `health` must surface connector auth state.
2. Retry/backoff/circuit behavior must be testable for HTTP and gRPC transports.
3. Timeout/limit/unavailable errors must be mapped consistently across transports.
4. Degradation behavior must be observable and must not violate CT2 semantics.

### 4.3 Negative checks

1. Broken bridge endpoint must fail fast (readiness gate effective).
2. Unsupported connector capability must emit explicit mapped error, not silent fallback.
3. Snapshot expansion path must prove connector call evidence in logs/events.
4. Invalid/expired cursor behavior must be explicitly tested and mapped.

## 5. Risks And Guardrails

1. **Risk:** Runtime keeps hidden legacy fallback and creates false green.  
   **Guardrail:** Add explicit assertions in CT2 scripts/log checks for connector call evidence.
2. **Risk:** HTTP/gRPC payload drift causes transport-specific behavior.  
   **Guardrail:** Contract fixture tests with shared golden payloads.
3. **Risk:** CI check-run naming drift causes pending required checks.  
   **Guardrail:** Freeze workflow job names in this plan and update branch protection together.
4. **Risk:** Platform integration completes, but real app still cannot be onboarded.  
   **Guardrail:** Add mandatory real-app pilot gate (V3-09) before declaring v0.3 ready.

## 6. Definition Of Done (v0.3 Connectorization)

1. V2 contract is frozen and implemented in SDK/runtime.
2. Snapshot/live/action/prewarm/health all route through V2 connector for in-process, HTTP, and gRPC paths.
3. Required gates are green with no legacy fallback masking.
4. Operational readiness gates (auth/ratelimit/retry/timeout/degrade) are green.
5. Runner semantics migration (`APPFS_V2_*` -> `APPFS_V3_*` aliasing and CI naming) is complete.
6. One real-app pilot passes sandbox validation with 24h evidence and rollback drill.
7. README + docs/v3 fully describe the shipping connector path and breaking changes.
